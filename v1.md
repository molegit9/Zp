# Zero-shot Phishing Detector (Hovering) v1.0 🚀

이 문서는 기존 피싱 탐지 시스템 아키텍처에서 현재 시스템으로 넘어오며 **개선된 핵심 변경점과 아키텍처에 대한 설명**을 담고 있습니다. 졸업작품 포트폴리오(논문/발표자료) 작성 시 참고할 수 있도록 학술적 관점에서 정리되었습니다.

---

## 1. 플랫폼 및 아키텍처의 패러다임 전환

*   **기존 시스템 (Naive LLM API Pass-through):**
    사용자가 마우스를 올린 URL 텍스트 자체를 아무런 가공 없이 거대 언어 모델(Gemini)에 그대로 전송하여 피싱 여부를 물어보는 단순 프록시(Proxy) 구조. 
    이로 인해 매번 무거운 텍스트 생성을 기다려야 했으며, LLM이 최신 도메인 정보나 오타 사칭 여부를 알 수 없어 환각(Hallucination) 오류에 취약했음.
*   **현재 시스템 (Metadata-Enriched Hybrid Inference):**
    로컬 브라우저에서 0.01초 만에 최상위 타겟 사칭 여부(Levenshtein)를 자체 계산하고, 서버가 0.3초 만에 국제 도메인 나이(RDAP)를 캐온 뒤, 구조화된 '위협 힌트 데이터' 형태로 AI에 주입시켜 초격차(1초 이내) 속도와 높은 확장성을 갖춘 전문화된 제로샷(Zero-shot) 아키텍처 구축.

## 2. 주요 기술 변경점 (What's New?)

### ① 클라이언트 연산 오프로딩 (Client-side Offloading)
*   **기존**: 오타 사칭 도메인(Typosquatting) 등 문자열 유사도 연산을 서버에서 처리.
*   **변경**: `Levenshtein Distance` 알고리즘을 크롬 프론트엔드(`content.js`) 내부로 완전히 이관시켜, 사용자의 로컬 환경에서 **0.01초 만에 최상위 타겟 도메인(구글, 네이버 등) 사칭 여부를 1차 판단**하도록 서버 병목 현상을 원천 차단함.

### ② 사전 메타데이터 주입 (Metadata Injection) 체계 도입
*   **기존**: 피싱 분석을 위해 URL 문자열 텍스트 하나만 덩그러니 AI에게 넘겨 전적으로 LLM의 추론이나 자체 지식 데이터에 의존함. (최신 웹사이트나 정교한 피싱을 판별 불가)
*   **변경**: 사칭 판별 결과와 도메인 생성 나이(Age)라는 가장 핵심적인 '오픈소스 위협(OSINT)' 팩트를 추출한 뒤, `Gemini 3.1 Flash Lite`의 프롬프트에 백그라운드 지식으로 미리 찔러넣어주는 구조(가벼운 RAG 형태)로 진화. AI는 복잡하게 텍스트를 생성하는 대신 제한된 문맥과 정보에 기반하여 즉각적인 점수 산출(Zero-shot) 연산만 수행하도록 경량화함.

### ③ 실시간 초고속 OSINT (RDAP) 통합
*   **기존**: 무겁고 느린 `Whois` 포트 통신을 사용하거나 해당 로직을 생략함.
*   **변경**: 국제 도메인 관리 기구의 표준 API인 **RDAP(Registration Data Access Protocol)** 통신을 비동기(`httpx.AsyncClient`)로 연결하여 0.5초 안에 도메인 나이(생성일)를 추출, 30일 미만의 사기성 도메인에 대한 치명적 감지력을 확보함.

### ④ 구글 최신 `google-genai` SDK 풀 비동기화 적용
*   **기존**: `google.generativeai` 구버전 V1Beta 통신 규격을 사용하여 실험적인 모델 연결 시 안정성 저하 및 404 모델 에러 발생.
*   **변경**: 최신식 원형 `google-genai` 컴포넌트로 전면 마이그레이션. 모델 응답을 순수 JSON 객체로 파싱하도록 강제하고, 비동기 호출(`client.aio.models`)을 사용하여 FastAPI 멀티 요청 시의 지연 시간(Blocking)을 해소함.

### ⑤ SQLite 실시간 캐싱 아키텍처 (Zero-Delay)
*   **기존 / 캐시 부재 시**: 같은 링크라도 다시 호버링할 때마다 AI에 요청하여 API 비용과 지연시간(1초) 낭비 발생.
*   **변경**: 분석이 한 번 끝난 결과는 로컬의 `security_logs.db`에 동기화. 사용자가 두 번째로 같은 링크에 호버링 할 때는 외부 접속 없이 로컬 DB에서 **0.01초** 만에 결과를 뽑아 렌더링하는 극강의 캐시 효율성을 달성함.

## 3. UI/UX 및 프롬프트 최적화 (보안 취약계층 특화)

**1. 점수 체계에 대한 인지적 혼란 제거**
기존 "위험 확률 10%" 과 같은 표기법을 폐기하고, **[안전도 (0~100점)] 체계**를 도입함. 
> 예: 100점(최고 안전, 초록 아우라), 10점(극위험 피싱 감지. 빨간 아우라 표시)

**2. 취약계층 맞춤형 응답 프롬프트 이식**
AI에게 "보안 취약계층 대상 전문가 역할"을 부여하는 페르소나 설계(Persona Design) 프롬프트를 삽입함. "SSL 인증서 위변조", "XSS 공격" 등의 난해한 단어들을 배제하고, "마인크래프트 공식 팬 커뮤니티입니다. 안심하셔도 됩니다" 같은 고령층/청소년 맞춤 일상 언어로 1~2문장의 툴팁 텍스트를 출력하도록 규격화. (졸업작품 페이퍼 작성 시 공공 서비스 관점에서의 사회 문제 해결 능력 어필 가능)

## 4. 최종 프레임워크 분석 파이프라인 (타임라인 요약)

1초(1000ms) 이내에 툴팁으로 결과를 응답하기 위해, 본 시스템은 아래와 같은 다중 메타데이터 병렬(Parallel) 파이프라인을 가동합니다.

*   **[0.0초] 판단 준비**: 사용자가 알 수 없는 링크 위에 마우스 커서를 올림 (0.2초 초과 대기 시 의도로 간주)
*   **(1단계) 문자열 유사도 비교 (Client-side Levenshtein)**:
    *   크롬 브라우저 로컬에서 `0.01초` 만에 즉시 실행. URL 도메인이 유명 브랜드(구글, 애플, 네이버 등)를 미세한 오타로 사칭했는지 수학적으로 평가 후 백엔드 전송.
*   **(2단계) 로컬 DB 지연시간 제로(Zero-Delay) 캐시 검증**:
    *   서버가 수신 즉시 로컬 SQLite(`security_logs.db`) 과거 기록을 조회. 이전에 누군가 검사한 흔적이 있다면 무거운 외부 분석 없이 **즉각적으로(0.01초)** 기존 결과를 반환하여 툴팁 표출 ⚡
*   **(3단계) 외부 보안 엔진 & OSINT 비동기 병렬 스캔**:
    *   초격차 속도를 위해, 백엔드가 **(A) VirusTotal 전 세계 블랙리스트 API 조회**와 **(B) RDAP 도메인 생성일자 추출(OSINT)**을 `asyncio.gather`를 통해 **동시에 병렬 방식**으로 찔러 수합 (0.3~0.7초 소요).
*   **(4단계) 메타데이터 통합 AI 추론 역량 (Gemini 3.1 Flash)**:
    *   앞선 [문자열 오타, VirusTotal 스캔 결과, 신규/안전 도메인 나이] 등 핵심 단서 객체(Metadata)를 프롬프트에 주입(RAG). AI는 URL 텍스트를 헤매는 대신 이 강력한 단서들을 바탕으로 **피싱 안전 점수(0~100점)**를 신속하고 오차 없이 산출(Zero-shot).
*   **(5단계) 프론트엔드 UI/UX 렌더링**:
    *   보안 취약계층도 100% 직관적으로 이해할 수 있는 "안전 점수"와 중학생도 이해할 수 있는 한 줄 해석(예: "공식 마인크래프트 페이지입니다")을 안전 등급 색깔과 함께 프리미엄 툴팁으로 렌더링. 🎈
